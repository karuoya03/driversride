Current Project: Bolt Interface Redesign

A comprehensive HCI project covering all six parts of our interface redesign process

Part 1: Project Definition

Problem Identification

Problem Overview

The Bolt ride-hailing application suffers from significant usability gaps where digital data (GPS pins and ETAs) often conflicts with physical reality. This misalignment creates cognitive load, coordination failures, and accessibility barriers for diverse user groups.

Bridging the Gulf of Evaluation and Execution through innovative interface design that transforms the Bolt ride-hailing experience into a collaborative partnership between users and technology.

Our project focuses on addressing GPS inaccuracies, communication breakdowns, and accessibility gaps in the current Bolt application through landmark verification, transparent system logic, and adaptive interfaces.

GPS Inaccuracies

Current Bolt interface showing GPS inaccuracies and location errors

Part 2: Contextual Research

Problem Analysis & User Research

Problem Statement

The Bolt application currently operates under a model where the system assumes its digital data (GPS pins and ETAs) is always correct, regardless of the physical world variables.

This creates a significant 'Gulf of Evaluation' when the digital state does not match the user's physical reality. Our project aims to redesign this interface into a 'collaborative partner'. By implementing landmark verification, transparent system logic and adaptive interfaces, the system will work with the user to overcome the GPS drift, communication breakdowns and accessibility issues, thereby reducing user anxiety and improving service efficiency.

User Characteristics

  • Commuters: Individuals with varying levels of technical proficiency
  • New users: Individuals unfamiliar with ride-hailing logic who require scaffolded guidance
  • Users with disabilities: Individuals requiring assistive UI features
  • Drivers: Secondary users who rely on accurate coordinates

Analysis of Current System

Map Inaccuracy

GPS shows incorrect location, pin drifts away from actual position, rural roads not recognized

Inconsistent Feedback

ETA jumps from 2 to 6 minutes without explanation, confusing users about arrival time

Communication Failures

Call button fails with weak signal, no alternative communication methods provided

Lack of Guidance

New user interface shows all features at once, overwhelming first-time users

Accessibility Issues

Small buttons, low contrast text, complex maps difficult for users with disabilities

Deficiencies Identified:

  • Map inaccuracy (Gulf of execution): Incorrect locations or failure to recognize rural roads
  • Opaque feedback (Gulf of evaluation): ETA changes without explanation
  • Communication fragility (Gulf of execution): Single voice-call channel fails in low network
  • Static interface (Accessibility Gap): One-size-fits-all approach

Mental Model vs. Conceptual Model Analysis

The core usability issues stem from the mismatch between the user's mental model (their understanding of how the system works) and the system's conceptual model (its actual architecture and logic).

Mental Model Diagram - Redesigned mental model reduces Gulf of Evaluation and Gulf of Execution

Conceptual Model Diagram - Changes from "App as Infallible Authority" toward "App as Collaborative Partner"

Key Model Disconnects:

  1. GPS Precision Disconnect: Users assume pin-drop accuracy; system has 10-50 meter variance
  2. ETA Transparency Disconnect: Users see ETA as fixed; system calculates it dynamically based on multiple variables
  3. Communication Disconnect: Users expect human-like coordination; system uses automated protocols
  4. Intelligence Disconnect: Users assume system understands landmarks; system only processes coordinates

Our redesign strategy focuses on making the system's conceptual model more visible and aligning it with user expectations through progressive disclosure and collaborative verification mechanisms.

Proposed Redesign Mechanism

Landmark Verification

Landmark-based pickup verification system using nearby physical landmarks

Transparent ETA

ETA with explanatory feedback showing reasons for time changes

  • Landmark Verification: Move from GPS dependency to physical landmark anchors
  • Progressive Transparency: Add reasoning layers to ETA changes
  • Fail-safe Communication: Multi-channel communication hub
  • Adaptive Interface: Flexible UI changes for accessibility

Task Analysis

Task Analysis Flowchart

Structured task analysis flowchart showing user interaction steps

Structured Task Analysis:

  1. Initialize the system (Open the app)
  2. Establish context (Progressive disclosure for new users)
  3. Plan your trip (Enter destination, verify drop-off)
  4. Pickup with landmark verification
  5. Monitor progress with transparent feedback
  6. Establish fail-safe communication

Usability Criteria

  • Spatial accuracy: Reduction in pickup error distance
  • Transparency: ETA changes accompanied by reasons
  • Inclusivity: Accessibility for users with disabilities
  • Cognitive load: Reduction in time-to-task for new users

Part 3: Design Alternatives

Three Design Approaches: Baseline, Transitional, and Final Redesign

Design Alternatives Overview

We developed three design alternatives to explore different approaches to solving the identified usability issues. Each alternative builds upon the previous one, incorporating user feedback and HCI principles to arrive at the final redesign.

Design Alternative 1: Existing Bolt Interface (Baseline)

This design represents the current Bolt interface and serves as a baseline for identifying current usability issues.

Existing Bolt interface showing the baseline design with identified issues

Observed Issues
  • GPS-only pickup pin causes map inaccuracy
  • ETA changes without explanation (Gulf of Evaluation)
  • Single voice-call communication channel
  • No visible onboarding or accessibility controls

Design Alternative 2: Intermediate Redesign (Transitional Design)

Its purpose is to explore early solutions, evaluate trade-offs, and identify remaining gaps before arriving at the final design.

This design improves usability compared to the baseline system but intentionally does not fully resolve all issues, allowing for iterative refinement based on HCI principles.

Transitional design showing early improvements while maintaining some limitations

Key Features Introduced
1. Semi-Improved Pickup Selection
  • Users can slightly adjust the GPS pin
  • A short text field allows users to describe their pickup location (e.g., "near the gate")
Improvement: Reduces some pickup errors compared to a fixed blue pin.
Limitation: Still relies heavily on user typing and does not fully align with physical landmarks.
2. ETA Indicator with Visual Warning
  • ETA changes are accompanied by a generic warning icon (âš ï¸)
  • No textual explanation is provided
Improvement: Signals that something has changed in the system.
Limitation: Users still do not understand why the ETA changed, leaving the Gulf of Evaluation unresolved.
3. Manual Communication Options
  • Both "Call Driver" and "Chat Driver" buttons are available
  • User must manually switch when one fails
Improvement: Adds redundancy compared to voice-only communication.
Limitation: Manual switching increases cognitive load, especially in low-network situations.
4. Basic Onboarding Hint
  • A single tooltip appears on first launch (e.g., "Tap the pin to adjust pickup")
Improvement: Provides minimal guidance for first-time users.
Limitation: Guidance is shallow and disappears quickly, offering no progressive learning.
HCI Improvements Achieved
  • Reduced execution errors compared to the baseline
  • Slightly improved visibility of system state
  • Partial reduction in user frustration
Remaining Usability Issues
  • Pickup location still mismatches real-world landmarks
  • ETA feedback remains opaque
  • Communication recovery is user-dependent
  • Accessibility options are still hidden or absent
Why Design Alternative 2 Was Not Selected as Final

Although Design Alternative 2 improved usability over the baseline system, it did not sufficiently reduce the Gulfs of Execution and Evaluation. Users still needed to interpret system behaviour, manually recover from errors, and adapt to a rigid interface. These shortcomings informed the final redesign decisions.

Design Alternative 3: Final Redesign

The redesigned interface includes four key improvements

The redesigned interface includes four key improvements:

  • Landmark Verification for Pickup: Select a nearby landmark to confirm your pickup point
  • Enhanced Communication Options: Multiple ways to contact the driver
  • Guided Onboarding for New Users: Step-by-step tips for first-time users
  • Accessibility Settings: Customizable options for accessibility needs
Key Improvements in Final Redesign
1. Landmark Verification for Pickup

HCI Principle: Match between system and real world

User Scenario: A commuter is waiting outside a busy shopping area where GPS accuracy is poor. The user opens the app and is prompted to confirm their pickup location using nearby landmarks.

Outcome: Pickup errors are reduced, coordination improves, and user confidence increases.

2. Transparent ETA Feedback

HCI Principle: Visibility of system status

User Scenario: A passenger notices their ETA has changed from 3 minutes to 6 minutes. The app displays "+3 min --- Heavy traffic detected."

Outcome: Users no longer assume the system is faulty or unreliable, maintaining trust and reducing anxiety.

3. Enhanced Communication Options

HCI Principle: Error tolerance

User Scenario: A user attempts to call the driver in a low-network area. The voice call fails, and the interface automatically presents alternatives like in-app chat.

Outcome: Coordination remains possible despite technical failures, preserving task continuity.

4. Guided Onboarding for New Users

HCI Principle: Recognition rather than recall

User Scenario: A first-time user opens the Bolt app. The interface highlights the pickup pin with a tooltip: "Tap here to set your pickup point."

Outcome: The learning curve is reduced, increasing adoption and early user satisfaction.

5. Accessibility Settings

HCI Principle: Inclusive design

User Scenario: A user with visual impairment opens the accessibility settings and enables high-contrast mode and larger text.

Outcome: Users with disabilities can complete tasks independently, promoting inclusivity and fairness.

What Was Refined or Dropped After Design Alternative 2
  • Removed free-text pickup descriptions due to ambiguity
  • Removed ETA warning icons without explanations
  • Removed manual communication switching
  • Removed single tooltip onboarding in favour of progressive guidance

Design Evolution Summary

The three design alternatives represent a journey from identifying problems (Alternative 1) to exploring partial solutions (Alternative 2) and finally arriving at a comprehensive redesign (Alternative 3). Each iteration was informed by HCI principles and user feedback, resulting in a final design that effectively bridges the Gulfs of Execution and Evaluation while promoting accessibility and user empowerment.

Part 4: Evaluation & Testing Plan

Prototype Evaluation Strategy

Evaluation Goals

The evaluation seeks to determine whether the redesigned system:

  • Improves alignment between digital location data and physical environment
  • Makes system behavior visible and interpretable
  • Maintains task continuity during failures
  • Reduces cognitive load for new users
  • Enables accessibility for users with disabilities

Testing Environment

Usability testing environment setup with controlled conditions

Evaluation Exercise 1: Usability Testing

Usability Testing Session

Task-based usability testing session with participants

Method: Moderated task-based usability testing

Participants: Commuters, new users, drivers

Metrics: Task completion rate, time, error rates

Rationale: Evaluates Gulf of Execution reduction

Evaluation Exercise 2: Comparative Interface Evaluation

Method: Comparative testing between existing and redesigned interfaces

Metrics: Pickup error distance, user preference, trust

Rationale: Isolates impact of redesigned features

Evaluation Exercise 3: Accessibility Testing

Method: Scenario-based testing under constrained conditions

Metrics: Success rate with adaptive UI, communication re-establishment time

Rationale: Evaluates inclusivity and error tolerance

Data Collection and Analysis

  • Quantitative data (time-to-task, error rates, completion success) analyzed using descriptive statistics
  • Qualitative data (user feedback, observed frustration, trust perception) analyzed thematically
